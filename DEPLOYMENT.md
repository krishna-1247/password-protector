# Production Deployment Guide

This document describes how to deploy the PDF Password Protector web application to production.

---

## Architecture Overview

- **Frontend**: Next.js (TypeScript) deployed to **Vercel** as a static site with client-side API polling.
- **Backend**: FastAPI (Python) deployed to **Railway** (or Render) as a Docker container.
- **State**: The application is **stateless**. No databases are required. All file storage is ephemeral (using `/tmp` or system-specific directory) and is automatically deleted after processing or when the 30-minute TTL expires.

---

## 1. Backend Deployment (Railway)

Railway supports direct Dockerfile builds, which makes it ideal for python backend deployments.

### Steps
1. Sign in to [Railway.app](https://railway.app).
2. Click **New Project** → **Deploy from GitHub repo**.
3. Select your repository.
4. Set the **Root Directory** settings to `/backend` (this ensures Railway builds the backend docker container, not the frontend root).
5. In **Variables** (Environment Variables), add:
   - `ALLOWED_ORIGINS`: `["https://your-frontend-domain.vercel.app"]` (Your Vercel deployment URL)
   - `MAX_WORKERS`: `4`
   - `JOB_TTL_MINUTES`: `30`
   - `CLEANUP_INTERVAL_SECONDS`: `300`
6. Click **Deploy**. Railway will build the container using `backend/Dockerfile` and assign a public domain (e.g. `https://your-backend.railway.app`).

---

## 2. Frontend Deployment (Vercel)

Vercel is optimized for building and serving Next.js applications.

### Steps
1. Sign in to [Vercel.com](https://vercel.com).
2. Click **Add New** → **Project**.
3. Import your GitHub repository.
4. Configure the Project:
   - **Root Directory**: Select `frontend`.
   - **Framework Preset**: Select **Next.js**.
   - **Build Command**: `npm run build`
5. Under **Environment Variables**, add:
   - `NEXT_PUBLIC_API_URL`: `https://your-backend.railway.app` (The backend URL generated by Railway)
6. Click **Deploy**. Vercel will build and serve your static web application.

## 3. Production Configurations & Tuning

### Upload Size Limits
By default, the backend limits file uploads to `500 MB` per request. If you want to increase or decrease this limit, update the `MAX_UPLOAD_MB` environment variable on Railway:
```env
MAX_UPLOAD_MB=1000  # Sets max upload to 1 GB
```

### Thread Pool Worker Size
Encryption is an CPU/IO bound task. We use a thread pool for parallel encryption. If you run the server on a machine with multiple cores, you can adjust the thread worker pool:
```env
MAX_WORKERS=8       # Adjust based on backend core counts
```

### Temporary File Expiry (TTL)
To ensure complete user privacy and avoid disk exhaustion, temporary folders are automatically deleted. You can tune how long they live after a job completes:
```env
JOB_TTL_MINUTES=15  # Ephemeral storage lifetime
```
